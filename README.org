#+TITLE: Multi-Agent Coding Workspace
#+AUTHOR: fanrado
#+OPTIONS: toc:2 num:nil

* Overview

A reusable tmux-based workspace for multi-agent coding sessions. A single
script call creates a fully laid-out terminal environment where four agents
work together — one human-controlled orchestrator and three autonomous Claude
Code agents — to plan, implement, test, debug, and validate features on
separate git branches.

The workspace is designed to be project-agnostic: point it at any directory
and it sets up the same consistent layout every time.

* Layout

One tmux session, one window, four panes arranged in a 2×2 grid:

#+BEGIN_EXAMPLE
┌─────────────────────┬─────────────────────┐
│                     │                     │
│    orchestrator     │     developer       │
│     (you)           │   (Claude Code)     │
├─────────────────────┼─────────────────────┤
│                     │                     │
│      tester         │     debugger        │
│   (Claude Code)     │   (Claude Code)     │
└─────────────────────┴─────────────────────┘
#+END_EXAMPLE

| Pane          | Branch            | Role                                               |
|---------------+-------------------+----------------------------------------------------|
| orchestrator  | =features/<name>= | You. Write plans as beads issues, dispatch phases  |
| developer     | =features/<name>= | Implements one plan phase at a time, no tests      |
| tester        | =test/<name>=     | Writes and runs unit tests for each new feature    |
| debugger      | =test/<name>=     | Fixes failing tests, iterates with tester          |

* Workflow

#+BEGIN_EXAMPLE
You (orchestrator, tab 1)
  │  1. Write plan-phase beads issues (each scoped to ≤ 2 files)
  │  2. Run: ./scripts/dispatch.sh <issue-id>
  ▼
Developer (tab 2)  [features/<name> branch]
  │  Reads the issue, implements the feature, commits
  │  If scope > 2 files → notifies you and waits for refinement
  │  On success → closes issue, runs sync to test branch
  ▼
Tester (tab 3)  [test/<name> branch]
  │  Detects new commits from sync
  │  Writes unit tests, runs them
  │  Creates a test-report beads issue (PASS or FAIL)
  ▼
Debugger (tab 4)  [test/<name> branch]
  │  If FAIL → fixes code, signals tester to rerun, iterates
  │  When all tests pass → syncs back to feature branch
  │  Creates a validation beads issue, notifies you
  ▼
You (orchestrator, tab 1)
     Review, close the validation issue to confirm, or reject
#+END_EXAMPLE

See =beads-workflow.org= for the full coordination protocol.

* Usage

** Start the workspace

#+BEGIN_SRC sh
./workspace.sh
#+END_SRC

Opens a session named =multiagents= rooted in the current directory.
The developer, tester, and debugger agents start automatically in their panes.

** Custom session name and directory

#+BEGIN_SRC sh
./workspace.sh -s myproject -d /path/to/repo
#+END_SRC

Or via environment variables:

#+BEGIN_SRC sh
SESSION_NAME=myproject WORKSPACE_DIR=/path/to/repo ./workspace.sh
#+END_SRC

** Attach to an existing session

#+BEGIN_SRC sh
./workspace.sh --attach
./workspace.sh -s myproject --attach
#+END_SRC

** Full option reference

#+BEGIN_EXAMPLE
Options:
  -s, --session NAME    Session name (default: multiagents)
  -d, --dir DIR         Working directory (default: current directory)
  -a, --attach          Attach to existing session if it exists
  -h, --help            Show this help
#+END_EXAMPLE

** Dispatching a plan phase

Once you have written a =plan-phase= beads issue, dispatch it to the developer:

#+BEGIN_SRC sh
./scripts/dispatch.sh <issue-id>
# example:
./scripts/dispatch.sh myproject-abc
#+END_SRC

The developer agent will see the issue, implement it on the =features/<name>=
branch, and trigger the rest of the pipeline automatically.

* Keybindings

| Key     | Action                             |
|---------+------------------------------------|
| =C-q=   | Kill the session and all its panes |
| =C-b d= | Detach from the session            |

Standard tmux navigation (=C-b= prefix) works as usual for switching panes,
resizing, scrolling, etc.

* Detaching and Reattaching

Detaching leaves the session and all agents running in the background.

** Detach

#+BEGIN_SRC sh
C-b d          # keyboard shortcut inside the session
tmux detach    # command line equivalent
#+END_SRC

** Reattach

#+BEGIN_SRC sh
tmux attach-session -t multiagents
tmux a -t multiagents            # short form
./workspace.sh --attach          # via the workspace script
#+END_SRC

** List running sessions

#+BEGIN_SRC sh
tmux ls
#+END_SRC

* File Structure

#+BEGIN_EXAMPLE
multiagents-setup/
├── workspace.sh               # Entry point — creates the tmux session and launches agents
├── beads-workflow.org         # Full beads coordination protocol
├── config/
│   └── workspace.conf         # Session name, window name, pane titles
├── agents/
│   ├── developer.md           # Developer agent instructions (CLAUDE.md)
│   ├── tester.md              # Tester agent instructions (CLAUDE.md)
│   └── debugger.md            # Debugger agent instructions (CLAUDE.md)
└── scripts/
    ├── tmux_helpers.sh         # Shared tmux functions (split, title, send)
    ├── dispatch.sh             # Dispatch a plan-phase issue to the developer
    ├── notify.sh               # Send an alert to the orchestrator pane
    ├── sync.sh                 # Sync between features/<name> and test/<name>
    └── agents/
        ├── developer.sh        # Launch developer Claude Code agent
        ├── tester.sh           # Launch tester Claude Code agent
        └── debugger.sh         # Launch debugger Claude Code agent
#+END_EXAMPLE

* Requirements

- =tmux= ≥ 2.6 (for =select-pane -T= pane title support)
- =bash= ≥ 4.0
- =claude= CLI (Claude Code) — installed and authenticated
- =bd= (beads) — issue tracker used for all agent coordination

** Beads quick reference

Run =bd prime= at the start of any session for full context.

| Command                    | Purpose                        |
|----------------------------+--------------------------------|
| =bd prime=                 | Load session context           |
| =bd ready=                 | Show issues ready to work on   |
| =bd create --title="..."=  | File a new issue               |
| =bd update <id> --claim=   | Claim an issue                 |
| =bd close <id>=            | Mark an issue complete         |
| =bd remember "insight"=    | Persist a note across sessions |
| =bd memories <keyword>=    | Search persistent memory       |

* Extending the Workspace

1. Add a new =PANE_*= variable in =config/workspace.conf=.
2. Add the corresponding split and title call in =workspace.sh=.
3. Create =agents/<role>.md= with the agent's instructions.
4. Create =scripts/agents/<role>.sh= to launch Claude Code with that file.
5. Wire the launch command into =workspace.sh= for the new pane.
